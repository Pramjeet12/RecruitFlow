## Database Migrations

**Alembic** manages schema changes. Migrations live in `backend/alembic/versions/`.

After modifying ORM models (in `backend/*/db_models.py`):
```bash
bash run.sh --migrate "describe_your_changes"
```

This command:
1. Runs `alembic revision --autogenerate` inside the container (detects schema diffs)
2. Copies the generated migration file to your host at `backend/alembic/versions/`
3. Applies it immediately

**Important:** Commit migration files to git so they're included in future Docker builds.

## Code Architecture

### High-Level Flow

**User Journey:**
1. Register/login via `/auth/login` → JWT token
2. POST `/api/v1/screening-jobs` with Drive folder link + job description + top-k
3. Backend creates a `ScreeningJob` record and spawns an async `orchestrate_job()` task
4. Browser opens an SSE stream (`/screening-jobs/{id}/events`) for real-time progress
5. When complete, GET `/api/v1/screening-jobs/{id}/results` returns top-k candidates ranked by GPT-4o score

**Worker Pipeline (Async):**
- Download all PDFs from Google Drive folder
- Create `CvResult` rows in DB (one per PDF, status=pending)
- Split CVs into **batches of 5** (configurable: `WORKER_BATCH_SIZE`)
- Run all batches **concurrently** via `asyncio.gather()`
- Each CV: extract text → find links (regex + PDF annotations) → async-fetch all links concurrently → send to GPT-4o for structured analysis → persist score + reasoning

**Key Design Patterns:**
- **Async/Await:** All I/O (httpx link fetching, DB, LLM API calls) is async
- **Background Task:** Job orchestration runs in a FastAPI background task, not a separate worker queue
- **SSE Polling:** Frontend polls job status every 1s via Server-Sent Events (no WebSocket overhead)
- **Structured Output:** OpenAI `beta.chat.completions.parse()` returns typed `CVAnalysis` (score, cv_data, reason)
- **Batch Processing:** CVs split into configurable-size batches; workers within a batch run sequentially, batches run in parallel

### Dependency Injection

See `backend/main.py` for wiring. Services are instantiated once and passed to controllers:
```
DatabaseServiceManager → UserModelService → UserServiceManager → UserRestController
                      → ScreeningJobModelService → ScreeningJobServiceManager → ScreeningJobRestController
                      → CvResultModelService → CvResultServiceManager → CvResultRestController
```

Controllers define FastAPI route handlers inside a `.prepare(router)` method.

### Modules

**`config/`** — Pydantic settings, loads `.env` from root or `backend/`

**`common/`**
- `logger.py` — JSON/pretty logging, correlation ID middleware, access log middleware
- `errors.py` — `AppError` hierarchy, FastAPI exception handlers
- `db_errors.py` — SQLAlchemy exception translator + `@handle_db_errors` decorator

**`database/`** — PostgreSQL connection pool, session context manager

**`auth/`** — JWT token creation/verification, OAuth2 dependency

**`user/`** — User model, registration, password hashing (Argon2)

**`screening_job/`** — Job model, manager (create/get/update_status/increment_processed), SSE controller

**`cv_result/`** — CV result model, bulk create, top-k query, stats (avg score, distribution, pass rate)

**`worker/`**
- `orchestrator.py` — Main async pipeline: download → batch → process → cleanup
- `cv_processor.py` — pdfplumber text extraction, regex link finding, async httpx link fetching (with truncation + timeout)
- `llm_client.py` — OpenAI GPT-4o call with structured output (CVAnalysis Pydantic model)
- `drive_client.py` — Google Drive HTML scraping (no API key needed; parses file IDs from public folder page) + per-file download
- `email_sender.py` — Gmail SMTP (requires `GMAIL_ADDRESS` + `GMAIL_APP_PASSWORD`)

**`static/`** — Single-page app: HTML/Tailwind/vanilla JS. No build step. States: login → register → form → progress (SSE) → results → history.

### Database Schema

```
users (id, email, full_name, hashed_password, is_active, created_at, updated_at)
screening_jobs (id, drive_link, folder_id, job_description, top_k, status, total_cvs, processed_cvs, error_message, created_by, created_at, updated_at)
cv_results (id, job_id, filename, drive_file_id, drive_preview_link, raw_text, links_data [JSON], score, cv_data, reason, candidate_email, status, error_message, created_at, updated_at)
```

Status fields: `pending`, `processing`, `completed`, `failed`, `cancelled`

### Configuration

See `.env.example`. Key variables:

**Database:**
- `POSTGRES_HOST=localhost` (set to `postgres` in Docker)
- `POSTGRES_PORT=5432`
- `POSTGRES_USERNAME=root`
- `POSTGRES_PASSWORD` (required)
- `POSTGRES_DATABASE=recruitflow`

**OpenAI:**
- `OPENAI_API_KEY` (required)
- `OPENAI_MODEL=gpt-4o` (fixed; not user-configurable)

**Worker:**
- `WORKER_BATCH_SIZE=2` — CVs per batch
- `LINK_FETCH_MAX_CHARS=1000` — Truncate fetched link content
- `LINK_FETCH_TIMEOUT_SECS=10` — HTTP timeout per link

**JWT:**
- `JWT_SECRET_KEY` (required; use a long random string)
- `JWT_ALGORITHM=HS256`
- `JWT_EXPIRE_MINUTES=1440` (24 hours)

**Gmail (optional, required only for send-mail feature):**
- `GMAIL_ADDRESS=you@gmail.com`
- `GMAIL_APP_PASSWORD=xxxxxxxxxxxx` (app-specific password)

**App:**
- `ENV=dev` (set to `prod` for JSON logging)
- `LOG_LEVEL=INFO`
- `TOP_K_OPTIONS=[5,10,15,20,25,30]` — valid top-k choices for the UI dropdown

| Method | Path | Auth | Notes |
|--------|------|------|-------|

| POST | `/api/v1/screening-jobs/{id}/cancel` | JWT | Cancel a pending/processing job |
| GET | `/api/v1/screening-jobs/{id}/results` | JWT | Top-k results: `{results: [{id, filename, score, cv_data, reason, drive_preview_link, candidate_email, ...}], top_k, job_id}` |

## Technology Stack

- **Framework:** FastAPI + Uvicorn
- **Database:** PostgreSQL 16 (Docker) / Supabase (prod)
- **ORM:** SQLAlchemy 2.x with Alembic migrations
- **Auth:** JWT (python-jose) + Argon2 password hash
- **AI:** OpenAI GPT-4o with structured output (`beta.chat.completions.parse`)
- **PDF:** pdfplumber (text + annotation extraction)
- **HTTP:** httpx (async, used for link fetching)
- **Drive:** gdown (for fallback) + direct HTML scraping (no API key needed)
- **Frontend:** Vanilla HTML/JS + Tailwind CDN (no build step)
- **Logging:** stdlib logging + custom JSON/pretty formatters + correlation IDs
- **Observability:** OpenTelemetry API (imported but minimally used; tracing spans exist)

## Important Implementation Details

### Google Drive Access
- **No API key required:** Drive folder must be publicly shared ("Anyone with link can view")
- **File listing:** Scrapes Drive HTML page to extract file IDs and filenames
- **Per-file download:** Constructs direct download URL: `https://drive.google.com/uc?id={file_id}&export=download`
- **Fallback:** If HTML scraping fails, uses `gdown.download_folder()` to download entire folder

### CV Processing Concurrency
- Links are fetched concurrently per CV using `asyncio.gather()`
- HTTP timeout: 10s per link (configurable)
- Content truncated to 1000 chars (configurable)
- Failed fetches return string: `"did not able to fetch"`
- LLM receives up to 6000 chars of CV text + all fetched link content in one prompt

### Job Cancellation
- Frontend calls `POST /cancel`; sets `status=cancelled`
- Worker checks `job.status` at the start of each batch
- If cancelled, current batch stops early; cleanup still runs

### Correlation IDs
- Every request gets a unique ID from header `X-Request-ID` (or generated)
- Propagated through logs via context var
- Returned in response headers and error responses

### Email Sending
- Requires Gmail app-specific password (not regular password)
- Synchronous SMTP (blocking); runs in thread via executor in orchestrator if needed
- Errors caught and returned in response; doesn't fail the entire send-mail endpoint

## Common Gotchas

1. **Drive folder must be public:** Private/restricted folders will fail with "No PDF files found"
2. **Migrations must be committed:** Auto-generated migration files in `backend/alembic/versions/` should be committed to git
3. **JWT expires:** Default is 24 hours; frontend must handle 401 and re-prompt for login
4. **OpenAI costs:** Each CV analysis costs money; batch size and timeouts affect total cost
5. **Link fetch timeouts:** Long-running sites (e.g., GitHub profile with thousands of repos) may timeout
6. **No worker process management:** Jobs run in the FastAPI event loop; if server crashes, in-flight jobs are lost (no persistence of partial progress)
